# Agent Debate（智能体辩论）

## 技能概述

当遇到有多个合理方案的设计决策时，启动多个独立的 Agent 分别从不同角度分析和辩论，
再由 Judge Agent 选出最优方案。

**核心思想：** 多角度独立分析 + 结构化辩论 > 单个 Agent 的判断。

## 适用场景

- 架构选型有多个可行方案（如队列选 Redis Streams 还是 RabbitMQ）
- API 设计选择（分页策略、数据格式等）
- 算法/性能权衡决策
- 两种及以上实现策略看起来都合理
- 用户明确要求"辩论一下"

**不适用：** 只有一个正确答案、决策无关紧要。

## 辩论流程（7 阶段）

```
1. IDENTIFY  — 明确议题，编写辩论简报（Debate Brief）
2. SIZE      — 根据决策的广度和深度计算需要多少 agent（2-5个）
3. CONFIRM   — 向调用方汇报 agent 数量、角度分配和调用成本，等待确认
4. PROPOSE   — 并行派出 N 个 agent，各自从不同角度提出方案
5. CRITIQUE  — 每个 agent 阅读所有其他方案，挑漏洞、找假设、挖风险
6. REBUTTAL  — 各自回应所有对自己的批评，有效则承认，无效则反驳
7. JUDGE     — 新 agent 通读全部记录，裁定胜负，给出实施方案
```

## Agent 数量计算

根据决策的**广度**和**深度**计算所需 agent 数量：

**广度（Breadth）**：有多少个独立可行的选项/范式？
- 1 分：二元选择（A vs B）
- 2 分：多个明确选项（A vs B vs C）
- 3 分：开放式设计空间

**深度（Depth）**：决策影响范围？
- 1 分：局部（单个模块/组件）
- 2 分：跨组件（多个服务或系统）
- 3 分：系统性（影响整体架构）

**公式**：`agent 数 = 广度分 + 深度分 - 1`，限制在 [2, 5]

## 确认闸门（Confirmation Gate）

启动辩论前，必须向调用方汇报并等待确认：

```
## Debate Plan
**Question:** <议题>
**Agent count:** N 个
**Angles:** <每个 agent 的论证角度>
**Estimated cost:** 3N+1 = X 次调用，4 轮

Proceed with this plan?
```

调用方可：
- 同意执行
- 覆盖 agent 数量
- 调整论证角度
- 修改评估标准优先级
- 直接指定方案跳过辩论

**调用方决定始终为最高优先级。**

## 成本

| Agent 数 | 总调用次数 |
|----------|-----------|
| 2 | 7 |
| 3 | 10 |
| 4 | 13 |
| 5 | 16 |

非重要决策不要启动辩论。

## 三种触发方式

| 触发方式 | 信号 | 说明 |
|---------|------|------|
| 计划检查点 | Plan 中标记 `[DEBATE: 议题]` | 执行到该步骤时启动辩论 |
| Agent 发起 | Agent 发送 `NEEDS_DEBATE: 议题` | 执行中遇到分歧，暂停请求辩论 |
| 用户发起 | 用户说"辩论一下" | 对当前决策点启动辩论 |

## Judge 输出

Judge 宣布赢家后会给出**实施方案**，可直接交给 writing-plans → executing-plans 技能落地。

## 关键规则

- 辩论简报（Debate Brief）必须在启动前写好，防止跑题
- 启动前**必须**经过确认闸门，向调用方汇报计划并获取同意
- Agent 数量由决策复杂度计算，不是固定 2 个
- Agent 输出在各阶段间**原文传递**，不得摘要
- 每个 Agent 分配**不同的论证角度**
- 如果双方方案趋同，**提前终止**，直接跳到 Judge
- HYBRID 判决必须**明确说明合并规则**
