# Agent Debate（智能体辩论）

## 技能概述

当遇到有多个合理方案的设计决策时，启动多个独立的 Agent 分别从不同角度分析和辩论，
再由 Judge Agent 选出最优方案。

**核心思想：** 多角度独立分析 + 结构化辩论 > 单个 Agent 的判断。

**自适应深度：** 对抗轮次后，测量方案间的结构分歧度。方案明显不同时走轻量 JUDGE，方案趋同时走深度 JUDGE（防止集体盲点）。按需分配 scrutiny——简单决策更快更省，复杂决策更深入。

## 适用场景

- 架构选型有多个可行方案（如队列选 Redis Streams 还是 RabbitMQ）
- API 设计选择（分页策略、数据格式等）
- 算法/性能权衡决策
- 两种及以上实现策略看起来都合理
- 用户明确要求"辩论一下"

**不适用：** 只有一个正确答案、决策无关紧要。

## 辩论流程（9 阶段）

```
1. IDENTIFY       — 明确议题，编写辩论简报（Debate Brief）
2. SIZE           — 根据决策的广度和深度计算需要多少 agent（2-5个）
3. CONFIRM        — 向调用方汇报 agent 数量、角度分配和调用成本，等待确认
4. PROPOSE        — 并行派出 N 个 agent，各自从不同角度提出方案
5. CRITIQUE       — 每个 agent 阅读所有其他方案，挑漏洞、找假设、挖风险
6. REBUTTAL       — 各自回应所有对自己的批评，有效则承认，无效则反驳
7. DIVERGENCE     — 轻量模型（Haiku）测量方案间结构分歧度，路由到合适的 JUDGE 深度
8. JUDGE          — 根据分歧度走快速/标准/深度 JUDGE，裁定胜负，给出实施方案
9. OUTPUT         — 将获胜方案写入 docs/debate/yyyy-MM-dd-{主题}.md，结果返回给调用方
```

## Agent 数量计算

根据决策的**广度**和**深度**计算所需 agent 数量：

**广度（Breadth）**：有多少个独立可行的选项/范式？
- 1 分：二元选择（A vs B）
- 2 分：多个明确选项（A vs B vs C）
- 3 分：开放式设计空间

**深度（Depth）**：决策影响范围？
- 1 分：局部（单个模块/组件）
- 2 分：跨组件（多个服务或系统）
- 3 分：系统性（影响整体架构）

**公式**：`agent 数 = 广度分 + 深度分 - 1`，限制在 [2, 5]

## 确认闸门（Confirmation Gate）

启动辩论前，必须向调用方汇报并等待确认：

```
## Debate Plan
**Question:** <议题>
**Agent count:** N 个
**Angles:** <每个 agent 的论证角度>
**Estimated cost:** 3N+2 到 3N+4 次调用（快速/标准: 3N+2, 深度: 3N+4）
Proceed with this plan?
```

调用方可：
- 同意执行
- 覆盖 agent 数量
- 调整论证角度
- 修改评估标准优先级
- 直接指定方案跳过辩论
- 覆盖 JUDGE 深度（fast/standard/deep）

**调用方决定始终为最高优先级。**

## 成本（自适应）

| Agent 数 | 快速/标准路径 | 深度路径（低分歧时） |
|----------|-------------|-------------------|
| 2 | 8 | 10 |
| 3 | 11 | 13 |
| 4 | 14 | 16 |
| 5 | 17 | 19 |

深度路径只在方案高度趋同（存在集体盲点风险）时触发，多数辩论走快速/标准路径。

## 自适应深度（分歧度路由）

对抗轮次（CRITIQUE+REBUTTAL）结束后，用一次 Haiku 调用测量方案间的结构分歧度，决定 JUDGE 深度：

| 分歧度 | 路径 | JUDGE | 说明 |
|--------|------|-------|------|
| 高（方案明显不同） | 快速 | 1 个 JUDGE | 对抗轮次已充分检验差异 |
| 中 | 标准 | 1 个 JUDGE | = 原先基线行为 |
| 低（方案趋同） | 深度 | 3 个独立 JUDGE + 盲审评分 | 趋同方案可能共享盲点，需要独立审查 |
| 分类器不确定 | 退回标准 | 1 个 JUDGE | 有疑问时用经过验证的默认值 |

**关键设计决策**：分歧度测量在对抗轮次**之后**，而非 PROPOSE 之后。仅在 PROPOSE 中看起来相似的方案，经过 critique 揭示不同假设、rebuttal 揭示不同防御策略后，结构上往往会分化。

## 三种触发方式

| 触发方式 | 信号 | 说明 |
|---------|------|------|
| 计划检查点 | Plan 中标记 `[DEBATE: 议题]` | 执行到该步骤时启动辩论 |
| Agent 发起 | Agent 发送 `NEEDS_DEBATE: 议题` | 执行中遇到分歧，暂停请求辩论 |
| 用户发起 | 用户说"辩论一下" | 对当前决策点启动辩论 |

## 产出与交付

1. **保存决策文档** — Judge 的完整决策（胜者、理由、实施方案）写入项目根目录 `docs/debate/` 下，命名格式 `yyyy-MM-dd-{主题slug}.md`
2. **返回给调用方** — 将文档路径和实施方案返回给调用方，由调用方自行决定下一步（可能是 writing-plans、superpowers:writing-plans、executing-plans 等任意 skill，或用户直接处理）
3. **仅需决策时** — 如用户只需辩论结果不需实施，保存文档即可停止

## 关键规则

- 辩论简报（Debate Brief）必须在启动前写好，防止跑题
- 启动前**必须**经过确认闸门，向调用方汇报计划并获取同意
- Agent 数量由决策复杂度计算，不是固定 2 个
- Agent 输出在各阶段间**原文传递**，不得摘要
- 每个 Agent 分配**不同的论证角度**
- 如果双方方案趋同，**提前终止**，直接跳到 Judge
- HYBRID 判决必须**明确说明合并规则**
- **分歧度检测在对抗轮次后运行**，不是 PROPOSE 后——防止误判收敛
- **Haiku 做结构相似度检测**（相似度测量，不是质量判断），全模型做推理
- **分类器置信度 < 50 时退回标准路径**——失败模式是"不是最优"，不是"错误的答案"
