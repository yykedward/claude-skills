# Design Debate Review（计划决策审查）

## 技能概述

在把计划或提示词提交执行之前，先丢进来扫一遍。这个 skill 会找出计划中隐含的、还没明确的决策点，然后返回：

1. **标注后的原文** — 计划/提示词中插入 `[Debate 决策主题]` 标记
2. **决策点清单** — 哪些需要辩论、为什么

然后由使用者决定哪些要辩、辩完再执行。

**核心原则：** 计划中的隐秘决策，执行前越早暴露，返工越少。

## 触发时机

- 写好了一段计划/提示词，提交前想找出决策漏洞
- 想在下本钱执行之前做一次压力测试
- 用户说"review this plan" / "审查计划" / "分析提示词中有什么决策点"

**注意：** 本 skill 不绑定任何固定流程位置，随时可以对任何计划或提示词使用。

## 流程（4 步）

### 1. REVIEW — 扫描决策点

仔细阅读计划/提示词，对照以下维度筛选争议点：

| 维度 | 判断标准 |
|------|----------|
| **对现有设计的影响** | 是否改变现有数据模型、API 契约、模块边界？ |
| **对未来开发的约束** | 选定后是否难以回退？是否会限制后续功能扩展？ |
| **多个可行方案** | 是否存在 2+ 个都合理但有不同 tradeoff 的方案？ |
| **隐性假设** | 计划是否依赖未经验证的假设？ |

**排除标准（不需要辩论）：**
- 只有一个明显正确的方案
- 项目已有既定惯例，无争议空间
- 决策影响范围极小（单个函数内部实现）

### 2. EXTRACT — 标注并提炼

对每个决策点，做两件事：

**a) 标注原文** — 在计划/提示词中决策产生的位置插入 `[Debate 决策主题]`：

```
原文：
系统用 Redis 缓存用户会话状态。

标注后：
系统用 Redis [Debate 会话缓存方案：Redis 单机 vs Cluster vs 本地内存] 缓存用户会话状态。
```

**b) 提炼辩论主题** — 每个主题一句话，具体、可回答：
- 不是"用什么数据库"，而是"用户余额用 MySQL 行锁还是 Redis Lua 脚本保证并发安全？"
- 各主题尽量独立（方便并行辩论）

标注依赖关系：

```
主题 A：<一句话>  [独立]
主题 B：<一句话>  [依赖 A 的结果]
主题 C：<一句话>  [独立]
```

### 3. OUTPUT — 返回结果

向用户呈现两部分：

**a) 标注后的原文** — 在原计划/提示词中内联 `[Debate 决策主题]` 标记。

**b) 决策点清单：**

```
## 识别到 N 个决策点

| # | 位置 | 决策主题 | 为什么需要辩论 |
|---|------|----------|----------------|
| 1 | 第X段 | <主题 A> | <一句话理由> |
| 2 | 第Y段 | <主题 B> | <一句话理由> |

**并行组：** 主题 1、3 可并行
**串行组：** 主题 2 依赖主题 1
```

然后询问用户是否执行辩论、跳过某些主题、或全部跳过。用户也可以调整主题措辞。

### 4. EXECUTE — 执行辩论（可选）

仅在用户确认后执行。由用户决定辩哪些、按什么顺序。

- **独立主题：** 并行调用 `yykedward-skills:agent-debate`，每个主题一个 debate
- **有依赖的主题：** 等待前置 debate 完成后，将其结论作为下一个 debate 的 context

每个 debate 的输出保存在 `docs/debate/` 下。

所有辩论完成后，汇总结果：

```
## 辩论结果汇总

| # | 主题 | 结论 | 关键理由 |
|---|------|------|----------|
| 1 | <主题> | <方案 X> | <一句话> |
| 2 | <主题> | <方案 Y> | <一句话> |
```

## 输出

核心产出是 **标注后的原文** + **决策点清单**。辩论执行是可选的，由用户驱动。

生成的 `docs/debate/` 文件由用户决定是否提交。
